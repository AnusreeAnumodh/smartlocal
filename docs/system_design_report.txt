====================================================================
1. SYSTEM DESIGN OVERVIEW
====================================================================

1.1 Objective
The system helps users:
- find trusted local service providers by category and city
- trigger SOS emergency alerts with current geolocation

1.2 Technology Stack
- Frontend: Angular 18 (feature modules, lazy loading, SCSS)
- Backend: Node.js + Express (REST API)
- Database: MongoDB (Mongoose ODM)
- Communication: JSON over HTTP

1.3 Architectural Style
- 3-tier architecture
  1) Presentation layer: Angular UI
  2) Application/API layer: Express routes + controllers
  3) Data layer: MongoDB collections (Service, SosAlert)

3. FLOW DIAGRAMS
====================================================================

3.1 Service Search Flow

User -> Dashboard UI -> LocalServicesService.getServices(category, city)
     -> ApiService GET /api/services?category=&city=
     -> Express Route (/api/services)
     -> service.controller.listServices()
     -> if DB connected: Service.find(query)
        else fallback seedServices filter
     -> JSON response (count, data, source)
     -> UI renders service cards/list

3.2 SOS Alert Flow

User clicks SOS
  -> Browser geolocation permission request
    -> if denied: UI shows "Location permission denied"
    -> if granted:
       -> LocalServicesService.triggerSos(payload)
       -> ApiService POST /api/emergency/sos
       -> emergency.controller.triggerSos()
       -> validate latitude/longitude
       -> if DB connected: SosAlert.create(...)
          else create fallback in-memory response
       -> return 202 Accepted + alert summary
       -> UI shows success/failure message

10. API SUMMARY
====================================================================

- GET  /api/health
  Returns service status, DB mode, and timestamp

- GET  /api/services?category={value}&city={value}
  Returns filtered service list sorted by verified and created time

- POST /api/emergency/sos
  Body: { userName, phone, latitude, longitude }
  Returns 202 Accepted with queued alert payload

11. DESIGN OBSERVATIONS AND NEXT DESIGN STEPS
====================================================================

- Current design is suitable for MVP and demo usage.
- Authentication is placeholder (demo token), not secure production auth.
- No explicit entity relationships yet (User, Provider, Review missing).
- For production-scale architecture, add:
  1) JWT-based auth + user roles
  2) provider verification workflow
  3) geospatial indexes for nearby search
  4) SOS dispatch integration (SMS/call/WhatsApp)
  5) audit logging and observability
