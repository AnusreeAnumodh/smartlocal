====================================================================
0. PROJECT OVERVIEW
====================================================================
SmartLocal is a full-stack web system that helps users find trusted local
service providers and trigger SOS emergency alerts using current location.

Tech stack:
- Frontend: Angular 18 + SCSS
- Backend: Node.js + Express
- Database: MongoDB + Mongoose
- API Style: REST + JSON

Core runtime capabilities:
- Search services by category and city
- Trigger SOS with geolocation
- Health monitoring endpoint
- Fallback mode when MongoDB is unavailable

====================================================================
1. HIGH-LEVEL ARCHITECTURE
====================================================================
Architecture type: 3-tier
1) Presentation Layer: Angular app (Auth + Dashboard modules)
2) Application Layer: Express routes/controllers
3) Data Layer: MongoDB collections

Main components:
- End User (Browser / Mobile Web)
- Angular Frontend (AuthGuard, ApiService, feature modules)
- Express Backend (/api/health, /api/services, /api/emergency/sos)
- MongoDB (services, sosalerts)

Key design decisions:
- Feature-based Angular modularization
- Controller-based backend business logic
- DB fallback startup mode for resilience
- Verified-first service sorting
- HTTP 202 for queued SOS acknowledgement

2. FLOW DIAGRAM SUMMARY
====================================================================
2.1 Service Search Flow
User -> Dashboard -> LocalServicesService -> ApiService -> Express Route
-> Controller -> MongoDB/Fallback -> JSON response -> UI list rendering

Request:
GET /api/services?category=<value>&city=<value>

Response:
{ count, data, source }

2.2 SOS Flow
User taps SOS -> browser geolocation -> POST /api/emergency/sos
-> backend validation -> DB create / fallback response -> user acknowledgement

Alternative paths:
- Geolocation unavailable
- Permission denied
- DB unavailable (fallback 202)
- API/network failure

7. SEQUENCE DIAGRAM (TEXT SUMMARY: SOS PATH)
====================================================================
Actors:
- User
- Frontend
- ApiService
- Backend
- MongoDB

Message flow:
1) User -> Frontend: tap SOS
2) Frontend -> ApiService: triggerSos(payload)
3) ApiService -> Backend: POST /api/emergency/sos
4) Backend -> MongoDB: create alert
5) MongoDB -> Backend: alert id/status
6) Backend -> Frontend: 202 Accepted
7) Frontend -> User: show result

Fallback branch:
If DB unavailable, backend returns fallback queued response with 202.
